METHOD OF DERIVING TEMPO AND POSITION IN A MUSICAL SEQUENCE
FROM A LIVE PERFORMER'S MIDI INPUT
==============================================================

Brief for prior-art search.  Inventor: Tony Weston, 2026.

----------------------------------------------------------------------
1. FIELD
----------------------------------------------------------------------

A music sequencer / tracker that advances through an authored note grid
under control of a human performer playing a separate live MIDI part,
deriving both the sequence's playback tempo and its row position in
real time from the timing and pitch of the performer's keystrokes.

Distinct from:
  - Fixed-BPM step sequencers (tracker, drum machine, DAW step sequencer).
  - MIDI-clock followers (slave to an external clock signal).
  - Tap-tempo (tempo set occasionally by manual taps, not continuously).
  - Score-following / audio-to-score alignment (e.g. Antescofo) which
    operates on a continuous audio stream, not on authored per-row cues
    in a step-sequencer grid.
  - Auto-accompaniment systems that detect chord voicings and play
    pre-canned backing patterns at a fixed or tap-set tempo.

----------------------------------------------------------------------
2. DATA MODEL
----------------------------------------------------------------------

The composition is a conventional multi-column tracker grid: rows are
discrete time steps; columns are independent voices / MIDI channels.

ONE column is reserved as the INPUT COLUMN.  It does not produce MIDI
output.  Each cell in the input column may carry a pitch (or "any
pitch") plus one of three role attributes:

  WAIT   - gating marker.  Sequencer halts when its playhead reaches
           this row.  Playback resumes only when the performer plays a
           live MIDI note whose pitch class matches the marker.

  SYNC   - tempo-lock marker.  Sequencer plays through the row
           normally, but a matching performer note at or ahead of this
           row triggers a snap event (see below) and re-derives the
           BPM.

  NOISE  - "expected but ignored" marker (no effect attribute).  A
           matching performer note is consumed without action: no
           snap, no tempo update, no MIDI output.  Used to filter
           expected decorative / repeated performer notes that should
           NOT drive the sequencer.

The remaining columns hold conventional sequenced notes that the
sequencer plays out as MIDI when its playhead reaches each row.

----------------------------------------------------------------------
3. PLAYBACK AND SNAP-FORWARD
----------------------------------------------------------------------

The sequencer's playhead advances row-by-row at the current derived
BPM.  When a performer note arrives, the system walks the input column
forward from the current playhead position to find the first cue (WAIT
or SYNC marker) whose pitch class matches the performer's note.  On a
match the system performs a SNAP:

  - The playhead is repositioned (jumped) to the cue row.
  - The MIDI output for that row is emitted immediately (sub-
    millisecond from key-down to MIDI-out).
  - The BPM estimate is updated (see Section 4).

This means that if the performer plays the cue note BEFORE the
sequencer has naturally ticked to it, the sequencer jumps forward to
meet the performer.  The performer therefore drives the sequencer's
row position directly via keystroke timing.

If the playhead is already halted at a WAIT row, a matching performer
note resolves the halt and the sequencer resumes playing from that
row.

If no matching cue is found in the rest of the pattern (the walk is
circular and may wrap past the end of the pattern back to the start),
the performer note is ignored.

----------------------------------------------------------------------
4. TEMPO DERIVATION
----------------------------------------------------------------------

Each snap event records (time_now, absolute_row_counter).  BPM is
computed from the delta between the current and previous snap:

    BPM = K * (delta_row) / (delta_time)

where K is a constant tying tracker rows to musical beats.  The value
becomes the BPM used to schedule subsequent natural row advances until
the next snap event re-derives it.  BPM is clamped to a song-defined
min/max.

The first snap after song start uses an initial BPM (also authored
per-song); subsequent snaps converge the BPM to the performer's
actual timing.

----------------------------------------------------------------------
5. NOISE-NOTE FILTERING (the distinctive mechanism)
----------------------------------------------------------------------

In live performance the performer is typically playing an independent
musical part (e.g. a bass line) that contains repeated or passing
notes that are NOT intended as sync cues.  Example: a 16-row phrase
where the performer plays the note C four times per phrase, but only
the 1st and 3rd C are intended as anchors.

Without filtering, every C the performer plays would match a future
WAIT marker and trigger snap-forward, causing the sequencer to race
ahead of the performer by one or more bars.

NOISE markers solve this.  The composer places a NOISE marker at each
row where a non-anchor performer note is expected.  The walk algorithm
then becomes:

  - WAIT or SYNC encountered, pitch matches  -> snap (Section 3).
  - WAIT or SYNC encountered, pitch mismatch -> walk stops, no action.
  - NOISE encountered, pitch matches AND not-yet-consumed
                                             -> mark this NOISE as
                                                consumed; absorb the
                                                performer note (no
                                                snap, no tempo update,
                                                no output); stop walk.
  - NOISE encountered, already consumed      -> skip; continue walk.
  - NOISE encountered, pitch mismatch        -> skip; continue walk.

Already-consumed NOISE markers are skipped on subsequent walks so that
a later performer note (e.g. the actual anchor) can still reach the
real WAIT/SYNC even when the playhead has not yet ticked past the
NOISE marker.

The consumed state for NOISE markers is implicitly reset whenever the
playhead position decreases (pattern wrap, manual jump, key-change,
WAIT timeout, seek) or the current pattern changes - i.e. each fresh
forward pass through a pattern starts with all NOISE markers
unconsumed.

----------------------------------------------------------------------
6. WORKED EXAMPLE
----------------------------------------------------------------------

Input column over a 16-row pattern, performer playing a bass-line of
four C notes per phrase:

  row  0:  WAIT  C
  row  4:  NOISE C
  row  8:  WAIT  C
  row 12:  NOISE C

  Event                  Playhead before     Sequencer action after
  ---------------------  ------------------  -----------------------------
  1st C from performer   halted at row 0     snap, play row 0, advance
  2nd C from performer   playing, row ~3     NOISE@4 consumed; no output
  3rd C from performer   playing, row ~5     NOISE@4 already consumed,
                                             skipped; snap to row 8;
                                             BPM = K * 8 / delta_t
  4th C from performer   playing, row ~9     NOISE@12 consumed; no output
  (tick wraps  15 -> 0)  end of pattern      consumed-NOISE state cleared
  next 1st C             halted at row 0     snap, play row 0...

Without the NOISE markers, the 2nd C would have been interpreted as
the anchor cue for WAIT@8 and the sequencer would have snapped forward
two rows ahead of the performer - i.e. a desync.

----------------------------------------------------------------------
7. ELEMENTS THE SEARCH SHOULD FOCUS ON
----------------------------------------------------------------------

The novelty claim is the COMBINATION of all of the following operating
as a single integrated mechanism.  The headline inventive structure is
the ROLE-TYPED GATING COLUMN; the tempo-derivation behaviour is a
consequence of that structure rather than the standalone novelty (see
Section 8 - Vivace / B-Keeper for prior art on the bare "tempo from
performer timing" idea).

  (a) An authored per-row "gating column" in a step-sequencer grid in
      which each cell carries a pitch (or "any-pitch" wildcard) plus
      one of three discrete ROLE attributes:

         WAIT   - halt the playhead at this row until a pitch-matched
                  performer note arrives, then resume.
         SYNC   - leave playback running but treat a pitch-matched
                  performer note at or ahead of this row as a tempo-
                  resync event.
         NOISE  - absorb a pitch-matched performer note silently;
                  produce no snap, no tempo update, no MIDI output.

  (b) Stateful consumed-marker tracking on NOISE cells: once
      absorbed, a NOISE marker is flagged as used and is skipped on
      subsequent walks until the playhead naturally ticks past it or
      the pattern resets.  This allows several NOISE markers in a
      single section to absorb several performer notes in order, and
      allows a later pitch-matched performer note to reach the next
      real WAIT/SYNC even before the playhead has ticked past the
      absorbed markers.

  (c) A real-time walk algorithm that, on every performer MIDI
      keystroke, scans the input column from the current playhead
      position forward (wrapping past end-of-pattern back to the
      beginning), classifies each encountered cell by its role, and
      either:
         - snaps the playhead immediately to a pitch-matched
           WAIT/SYNC row and emits that row's MIDI output, or
         - absorbs the keystroke against an unconsumed pitch-matched
           NOISE marker, or
         - skips a non-matching marker (or already-consumed NOISE)
           and continues walking to find the next cue.

  (d) Snap-forward with sub-millisecond key-to-MIDI-output latency:
      when the walk finds a forward WAIT/SYNC cue, the playhead is
      jumped (potentially across many rows or past end-of-pattern)
      and the new row's MIDI output is emitted before any other
      processing.

  (e) Tempo derivation as a CONSEQUENCE of the gating-column
      mechanism: the time delta between successive snap events drives
      a continuously updated BPM estimate that schedules natural row
      advancement until the next snap re-derives it.  (See Section 8
      - the bare idea of deriving tempo from performer event timing
      is not itself novel; the novelty is its integration with
      (a)-(d) above.)

----------------------------------------------------------------------
8. PRIOR-ART CATEGORIES TO RULE OUT
----------------------------------------------------------------------

Categories likely to surface but believed to NOT cover the combination
above:

  - MIDI clock master/slave: tempo from a continuous clock signal,
    not from per-event keystroke timing.
  - Tap-tempo: tempo set by occasional manual taps, no per-row
    gating, no snap-forward, no noise filter.
  - Auto-accompaniment (Yamaha QY-series, Band-in-a-Box, OneTouch,
    Korg Pa-series styles): chord-recognition driving pre-canned
    accompaniment loops; tempo is fixed or tap-set, not derived from
    keystroke inter-arrival.
  - Score-following / audio-to-score alignment (IRCAM Antescofo,
    Music Plus One, Cont's research): operates on continuous
    pitch-detected audio, performs HMM-style alignment, not on
    authored per-row gating cells with snap-forward semantics.

  - Coda Music "Intelligent Accompaniment" (Vivace / SmartMusic
    family; WO1994028539A2 and continuations).  CLOSEST PATENT
    PRIOR ART for tempo derivation.  Derives tempo from time delta
    between matched performer events and expected-score events,
    using tolerance windows and skip thresholds for missed notes.
    Distinct from the invention because it operates over a
    continuous performance score rather than a step-sequencer
    grid; it has no equivalent of WAIT (composer-authored halt) or
    NOISE (composer-authored absorption marker) and no role-typed
    gating-cell column.  The patent family should be obtained in
    full and the claims compared element-by-element.

  - B-Keeper (Robertson & Plumbley, Queen Mary University of
    London, 2007 NIME paper; extended in Computer Music Journal
    37(2), 2013).  CLOSEST ACADEMIC PRIOR ART.  Real-time beat
    tracker that re-tempos a sequencer from a live drummer's
    kick-drum onsets.  No authored cue rows, halt/resume, or
    noise filter; effectively a continuous beat-track-driven
    clock follower.  Cite to demonstrate that "sync sequencer to
    performer" is a known goal; novelty rests on the gating-cell
    mechanism, not the goal.

  - Visual / gestural conductor systems (US20130032023A1,
    US5275082, "Visual Conductor" device): tempo follows a
    conductor's hand or baton motion captured via camera or
    accelerometer.  No grid, no per-row cues, no halt/resume.
  - Step-sequencers with conditional rows (Elektron probability /
    parameter locks, NerdSEQ conditional trigs): conditions are
    deterministic / probabilistic per row, not driven by external
    keystroke timing.
  - Live-loopers with quantize-on-arm: align loop boundaries to a
    grid; no per-row gating; tempo is grid-derived not keystroke-
    derived.
  - DAW input-quantize / human-quantize features: post-hoc snap of
    recorded MIDI events to a grid; not real-time control of a
    sequencer's playback row by performer input.
  - Karaoke / follow-along scrolling-score systems: visual cue, not a
    live MIDI sequencer being driven by performer input.

----------------------------------------------------------------------
9. TERMINOLOGY USED IN THIS DOCUMENT
----------------------------------------------------------------------

  Tracker / step-sequencer: software that plays back a 2D grid of
    notes (rows = time steps, columns = voices) by stepping through
    rows at a fixed or programmed rate.

  Pattern / block: one section of the grid, typically 16, 32, or 64
    rows long, played back as a unit.

  Row: one time step in a pattern.  Holds at most one cell per
    column.

  Cell: one note slot at a specific (row, column).  Contains a pitch
    and optional effect / role attribute.

  Pitch class: the pitch modulo 12 (i.e. octave-independent note
    name: C, C#, D, ...).

  Playhead: the sequencer's current row position within the active
    pattern.

  Snap: a real-time repositioning of the playhead in response to a
    performer keystroke, plus the immediate MIDI output of the new
    row.

  Walk: the search through the input column from the current
    playhead position outwards, looking for a matching cue.

  Consumed (of a NOISE marker): flagged as having absorbed a
    performer keystroke; will be skipped on subsequent walks until
    the consumed state is reset.

End of brief.
